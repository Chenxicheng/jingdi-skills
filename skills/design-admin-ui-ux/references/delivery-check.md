# 交付检查

仅在调用包含 `delivery-check` 参数时读取并执行本文件。

## 参数解析

- `delivery-check`：以当前请求涉及的全部 UI/UX 变更为检查范围。
- `delivery-check=<path|page|component>`：以参数值指定的文件、页面或组件为检查范围。

参数值无法解析时，使用当前请求中可确认的最小 UI/UX 变更范围，并在报告中标明范围假设。

## 检查步骤

### 1. 固定范围

列出纳入检查的页面、组件、状态和用户路径。读取其 project 规则、design token、组件封装、同类实现与相关测试。

完成标准：每个修改过的用户可见元素和交互路径均已纳入，未修改区域不扩张检查范围。

### 2. 逐项验证

对每项给出 `通过`、`未通过` 或 `不适用`，并记录可复查证据。

#### 任务与结构

- [ ] 主任务、用户角色和成功结果清晰
- [ ] 页面结构、导航和操作顺序符合用户任务
- [ ] 每个操作上下文只有一个清晰主操作
- [ ] 页面没有重复入口或无任务价值的模块

#### Project 一致性

- [ ] 复用 project 既有组件、design token、图标和术语
- [ ] 颜色来自 project 语义 token，状态不只依赖颜色
- [ ] 布局、间距、字体、层级和动效符合 project 约定
- [ ] 与同类页面的行为和反馈保持一致

#### 状态与反馈

- [ ] 正常、加载、空、错误、无权限、禁用、提交中和成功状态已覆盖
- [ ] 用户操作获得就近、及时且不过量的反馈
- [ ] 长任务提供进度、取消或转后台能力
- [ ] 并发变化和失败保留用户输入及恢复路径

#### 表单与数据

- [ ] Label、帮助文本、格式、单位和限制清晰
- [ ] 校验错误可定位、可理解、可修复
- [ ] 表格和数据格式支持扫描、比较和极端数据
- [ ] 敏感信息按 project 安全策略展示和脱敏

#### 风险与权限

- [ ] 高风险操作说明具体对象和后果
- [ ] 可逆操作提供撤销，不可逆操作提供匹配风险的确认或安全校验
- [ ] 权限受限状态提供合适说明且不泄露无知情权限信息
- [ ] 返回、取消、失败和离开页面均有可控路径

#### 可访问性与适配

- [ ] 键盘操作、焦点顺序和浮层焦点恢复正常
- [ ] 语义结构、Label、状态消息和可访问名称完整
- [ ] 文字与非文字对比度符合 WCAG 2.2 AA 或 project 更高标准
- [ ] 缩放、长文本、窄窗口和减少动态效果偏好可用
- [ ] 触控目标和目标间距符合 WCAG 与 project 约定

#### 验证证据

- [ ] 适用的 test、lint 和 build 已通过
- [ ] 关键路径已人工或自动验证
- [ ] 视觉变化已在目标窗口和主题下检查
- [ ] 未通过项均给出最小修复建议

完成标准：所有检查项均有结论；每个未通过项均有位置、用户影响、证据和最小修复建议。

## 报告格式

```markdown
# UI/UX 交付检查

检查范围：<path、page 或 component>
总体结论：<通过 | 有条件通过 | 不通过>

## 未通过

- [级别] <位置>：<问题>
  - 用户影响：<影响>
  - 证据：<可复查证据>
  - 最小修复：<建议>

## 通过

- <检查项>：<证据>

## 不适用

- <检查项>：<原因>

## 验证

- `<command>`：<结果>
```

级别使用：

- `阻断`：任务无法完成、结果错误、安全或可访问性底线失守
- `重要`：显著增加认知、操作或恢复成本
- `优化`：不阻断任务，但存在明确且可验证的体验改进
